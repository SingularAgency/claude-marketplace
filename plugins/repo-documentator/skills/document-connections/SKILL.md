# document-connections

Regenerates the connection map from code. Standalone refresh of `docs/connections.md` and each `docs/modules/*/connections.md`. Useful after big refactors or whenever the user wants to re-derive the architecture view.

## Core principle

**No questions to the user.** Everything is derived from the code. The user reviews the output.

## Pre-conditions

- `docs/` exists (`document-init` has run)
- Modules defined under `docs/modules/`

## Flow

### Step 1: load the module list
Read `docs/modules/` to get the module list and their `paths`. If a module's `paths` are missing, infer them from the module name.

### Step 2: run the analyst
Invoke `connection-analyst`:

```
Agent({
  description: "Map inter-module connections",
  subagent_type: "connection-analyst",
  prompt: "Analyze the technical connections between modules: {list}. For each one, read its code at {paths}. Detect: cross-imports, HTTP/RPC calls, pub/sub events, queues, webhooks, external API calls, and especially subtle couplings (shared DB, shared storage, env vars, implicit conventions). Cite evidence path:line. Generate matrix, global Mermaid diagram, and list subtle couplings with emphasis."
})
```

### Step 3: write `docs/connections.md`
Use the `connections-global.md` template. Fill from the agent's output:
- Global Mermaid diagram (graph LR)
- Dependency matrix
- External systems table
- Domain events (if any)
- Subtle couplings (verbatim from the analyst — these are the gold)
- Critical paths: derive heuristically as the longest sync chains touching external systems. Mark each candidate with `{{review}}` so the user can confirm or remove.

Frontmatter: `status: draft`, `auto-detected: true`, `last-reviewed: <today>`.

### Step 4: write each `docs/modules/<x>/connections.md`
Use the `module-connections.md` template per module. Fill:
- Local diagram (subgraph: this module + direct neighbors)
- "What it CONSUMES" table — sources, type, mechanism, data, criticality (`{{TBD}}` if not inferable from retry/fallback patterns)
- "What it EXPOSES" table
- Events emitted / subscribed
- Shared data / subtle couplings affecting this module
- "If a connection fails" table — fill `{{TBD}}` for any connection that has no visible retry/fallback in the code

### Step 5: flag potential ADRs
Append to `docs/_AUTOGENERATED.md` a list of ADR candidates:
- Each subtle coupling above a severity threshold (shared DB tables across >2 modules, shared mutable env vars, implicit payload conventions across services)
- Heavy outbound dependency on a single external SDK with no abstraction layer

The user reviews and decides whether to create ADRs. **Do not create ADRs autonomously** — those are decisions, not derivations.

### Step 6: open the pull request
Follow the shared PR flow at `${CLAUDE_PLUGIN_ROOT}/skills/document-init/references/pr-flow.md`. In short:
1. `git checkout -b business-docs/connections-<YYYYMMDD-HHMM>`
2. `git add docs/connections.md docs/modules/*/connections.md docs/_AUTOGENERATED.md`
3. Commit: `docs: regenerate inter-module connection map`
4. `git push -u origin <branch>`
5. `gh pr create` (or REST API with `GITHUB_TOKEN`).

If no credentials, fall back to local-only and print manual push/PR steps.

### Step 7: report
Print:
- **PR URL** or local path + manual steps
- Files written (count)
- Connections detected (count)
- Subtle couplings (short list)
- Critical path candidates (short list, marked `{{review}}`)
- ADR candidates surfaced

## Rules

- **No questions to the user.**
- **Cite evidence** (path:line) on every connection.
- **Subtle couplings with emphasis.** That's the most valuable output.
- **Mermaid line conventions:**
  - Solid (`-->`) = sync (HTTP/RPC)
  - Dotted (`-.->`) = async (events/queues)
  - Thick (`==>`) = critical/blocking
- **Don't mix** this doc with business docs. Architecture lives here, module purposes elsewhere.
- **Always open a PR** when credentials allow. Never push to the default branch.
- Operate in English by default.
