# document-workflow

Derives an end-to-end workflow doc from the code. Inputs: the workflow name and (optionally) an entry point. The agent traces the call chain, events, and state transitions, then writes the doc with **two diagrams**: a non-technical business view + a technical sequence diagram for engineers.

## Core principles

- **No questions to the user.** What the code can't tell us is `{{TBD}}`.
- **Two-layer diagrams.** Every workflow has a Business view (non-technical flowchart for PMs/stakeholders) AND a Technical sequence (for engineers). Put the business view first.
- **Non-technical prose for happy path + edge cases.** No `[API → DB]` notation, no path:line refs, no internal endpoint mentions in those sections.
- **Cite evidence in the technical sequence only.** That's the engineer-friendly view; let it be precise.

## Inputs accepted
- Required: workflow name (e.g. "process customer order")
- Optional: entry point hint (an endpoint, a function name, a CLI command, a file path)

If no entry point is provided, infer it: look for endpoint/handler names matching the workflow name, top-level CLI commands, scheduled jobs whose name matches, or queue consumers tagged with the workflow domain.

## Flow

### Step 1: locate the entry point
Search for the entry point using the workflow name. If multiple candidates exist, pick the one with the highest match-score and list the others in `docs/_AUTOGENERATED.md` for the user to review. Do not block.

### Step 2: trace the chain
Starting from the entry point, follow:
- Function/method calls (recurse, capped at depth 8 to keep it tractable)
- Event publications (`publish`, `emit`, `dispatch`, `send_message`, etc.) and consumers
- Queue enqueues and their consumers
- DB state transitions (status enums, lifecycle columns updated)
- External API calls
- Outbound webhooks and notifications

Record each hop with module, file:line, mechanism (sync HTTP / event / queue / DB write / external).

### Step 3: identify modules involved
Cross-reference each hop's location against `docs/modules/` to label the module it belongs to.

### Step 4: write the workflow file
Write `docs/workflows/<name>.md` following `${CLAUDE_PLUGIN_ROOT}/templates/workflow.md`. Sections (in order):

1. **Frontmatter** — `workflow`, `type: end-to-end`, `modules-involved: [...]`, `external-systems: [...]`, `last-reviewed`, `status: draft`, `auto-detected: true`, `owner: {{TBD}}`.
2. **Business objective** — what this workflow achieves from the business POV. Pull from PRD where possible.
3. **Trigger** — what kicks it off (human action, event, schedule).
4. **Actors involved** — humans + systems with one-line role each.
5. **Business view (non-technical)** — Mermaid `flowchart LR` with persona + outcome colors, plus three short paragraphs:
   - "What this is, in one paragraph"
   - "Who's involved"
   - "Why it matters"
   The flowchart must use plain-English nodes only. NO endpoints, NO databases, NO internal services.
6. **Technical sequence (for engineers)** — Mermaid `sequenceDiagram` with `autonumber`, all participants (human + system), full endpoint / DB / event detail.
7. **Variants** (if applicable) — branches that materially change the flow (e.g. agent-driven vs manual).
8. **Happy path** — numbered steps in **plain English**. NO `[API → DB]` notation, NO path:line refs.
9. **Edge cases** — each one in `What it looks like to the user / Why it happens / Resolution` format. Business framing.
10. **Business rules applied** — links to `modules/<x>.md` with rule restated in plain language.
11. **Key data flowing** (table) — plain English description of what information passes between actors.
12. **SLAs / time expectations** — real numbers if measurable; honest TBD otherwise.
13. **Metrics / KPIs** — what to measure for this workflow's health.
14. **What has historically failed here** — incident log if any; honest TBD if empty.

### Step 5: derive structure
- **Trigger**: the entry point (HTTP endpoint, scheduler, queue, CLI). Concrete.
- **Actors**: include human actors with role names where the code hints at them (admin endpoint → "admin user", customer-facing → "end user", scheduled → "no human").
- **Business flowchart**: 5-10 nodes maximum. Start with a person (`:::person`), end with an outcome (`:::good` or `:::neutral`). Use `:::warn` for queued / pending / cautionary states.
- **Technical sequence**: include `actor` declarations for humans, `participant` for systems. Use `note over X,Y:` for subtle couplings the engineer should be aware of.
- **Edge cases**: derived from `try/except`, error responses, conditional branches in the chain. Each one cited path:line **in the Technical sequence only**, not in the Edge cases prose section.

### Step 6: lint Mermaid
Both diagrams. Search for parens inside `|...|` edge labels and replace with em-dashes. Mermaid's parser interprets `(` as a node-shape delimiter and the diagram fails to render.

If the file already exists with user-edited content (no `auto-detected: true` on it), write to `docs/workflows/<name>.proposed.md` instead and tell the user.

### Step 7: cross-references
- Update `docs/README.md` — add the workflow to the critical workflows table.
- Update each involved `docs/modules/<slug>.md` `Related workflows` section — append a link with the role this module plays in this workflow.

### Step 8: open the pull request
Follow `${CLAUDE_PLUGIN_ROOT}/skills/document-init/references/pr-flow.md`. In short:
1. `git checkout -b business-docs/workflow-<name>-<YYYYMMDD-HHMM>`
2. `git add docs/workflows/<name>.md docs/README.md docs/modules/*.md`
3. Commit: `docs: derive <name> workflow from code`
4. `git push -u origin <branch>`
5. `gh pr create` (or REST API with `GITHUB_TOKEN`).

If no credentials, fall back to local-only and print manual push/PR steps.

### Step 9: report
Print:
- **PR URL** or local path + manual steps
- File written (path)
- Modules touched
- Edge cases auto-detected (count)
- Business rules referenced (count)
- Top `{{TBD}}` fields the user should fill (real edge cases the user knows that didn't show up in code, SLAs measured in production, incident history)

## Rules

- **No questions to the user.** What the code can't tell you is `{{TBD}}`.
- **Two diagrams, in this order:** Business view first, Technical sequence second.
- **Business view nodes must be plain English** — no endpoints, no databases.
- **Cite evidence (path:line) only in the Technical sequence.** Other sections stay non-technical.
- **Business flowchart edge labels MUST NOT contain parentheses** — use em-dashes. Mermaid breaks on `(text)` inside `|...|`.
- **If the workflow is huge** (>15 hops), split into sub-workflows automatically (one parent + N child files) instead of producing a giant unreadable diagram.
- **Always open a PR** when credentials allow.
- Operate in English by default.
