# document-module

Regenerates the docs for a single module from the code, autonomously. Useful after `document-init` if a specific module needs a deeper pass or has changed significantly.

## Core principles

- **No questions to the user.** Everything is derived from the code; what can't be derived is `{{TBD}}`.
- **Non-technical by default in business sections.** Capabilities, Business Rules, Known Risks, Module Flow, Related Workflows must read like a PM-friendly doc. NO file paths, NO line numbers, NO endpoints, NO SQL outside Data Model.
- **Single file per module.** Output is `docs/modules/<slug>.md` (one file). The 5-files-per-module structure (purpose / actors / business-rules / connections / known-pain) is **deprecated**.
- **Diff-merge.** Preserve any user-edited content (sections without `auto-detected: true` in frontmatter).

## Pre-conditions
- A `docs/modules/<slug>.md` file exists (created by `document-init`), OR you create it from `${CLAUDE_PLUGIN_ROOT}/templates/module.md`.
- The module's code paths are either in the existing frontmatter `paths:` field or detectable by the module name.

## Flow

### Step 1: locate the module
- Read the existing `docs/modules/<slug>.md` if it exists; pull `paths` from frontmatter.
- If absent, find the module by name (folder match, then file/symbol search).
- Build the file list of all source files belonging to the module.
- Read existing high-signal sources: repo `README.md`, any `docs/PRD*`, `docs/architecture*`, `CODEOWNERS`. Use them to fill `Purpose`, glossary candidates, and rule origins.

### Step 2: read the module's code
Use Read, Glob, Grep, Bash freely. Build an internal model:
- Public API (HTTP routes, CLI commands, exported functions, GraphQL resolvers)
- Models / entities / DB tables touched
- External SDKs and API calls (Stripe, AWS, GitHub, Slack, etc.)
- Cross-module imports (consumes / exposes)
- Numeric or role-based conditionals that look like business rules
- TODO/FIXME/HACK/@deprecated comments

### Step 3: regenerate the file (one cohesive doc)
Overwrite (or create) `docs/modules/<slug>.md` following `${CLAUDE_PLUGIN_ROOT}/templates/module.md`. Sections (in order):

1. **Frontmatter** — `module`, `last-reviewed: <today>`, `status: draft`, `auto-detected: true`, `owner: <git first-commit author for module paths>`, `paths: [...]`.
2. **Purpose** (2-3 paragraphs, plain language) — pull from PRD, repo README, package description.
3. **Module flow** — one Mermaid `flowchart LR` between Purpose and Capabilities. Persona/outcome colors. NO parens inside `|...|` edge labels.
4. **Capabilities** — plain-English bullets describing what users can DO. **NO file paths, NO line numbers, NO endpoint patterns, NO SQL.**
5. **Data Model** — full table schemas, FKs, constraints, indexes. Cite migration filenames.
6. **Connections** — what it consumes / exposes / publishes / subscribes. Cite path:line.
7. **Business Rules** — `Rule:`, `Owner:` (auto-detected first-commit author with verify caveat), `Origin:` (commit ref). NO `Implementation:`, `Revisable:`, or `Exceptions:` lines.
8. **Actors** — Human actors (real role inferences), External systems (specific), Other modules.
9. **User journey maps** — one Mermaid `journey` diagram per real actor. Each shows the actor's steps with realistic satisfaction scores (0-5). NO colons inside step text — Mermaid splits on colons; use em-dashes inside step text.
10. **Related workflows** — list of links to `workflows/<flow>.md` with role per workflow.

#### Diff-merge rules
- A field already filled in a section that is NOT marked `auto-detected: true` is **user content** — preserve it verbatim.
- Sections marked `auto-detected: true` get fully regenerated.
- Add a comment `<!-- auto-generated -->` near regenerated content if the file has mixed content.

### Step 4: emit a glossary candidate list
Append to `docs/glossary.md` under "Terms detected from code, definitions pending review" any new domain nouns extracted from this module's public API and models. Don't define them.

### Step 5: detect undocumented decisions
Scan recent commits to the module (`git log --oneline -n 50 <paths>`) for messages that look like decisions ("switch to", "replace X with Y", "migrate from"). List candidates in `docs/_AUTOGENERATED.md` as suggested ADRs. Don't create the ADRs automatically.

### Step 6: lint Mermaid
Two failure modes to lint for:
1. **Parens inside `|...|` edge labels in flowcharts** — Mermaid parses `(` as a node-shape delimiter and the diagram fails to render. Replace with em-dashes.
2. **Colons inside step text in `journey` diagrams** — Mermaid splits step lines on `:`. A step description containing an extra `:` confuses the parser. Replace inline colons with em-dashes inside step text.

### Step 6.5: update the state file
If `docs/.business-docs-state.json` exists, update only this module's entry:
- `modules[?slug=='<this>'].last_regenerated_commit = HEAD`
- `modules[?slug=='<this>'].last_regenerated_at = <now>`

Do **not** touch `last_run` (that belongs to `document-init` / `document-refresh`). Manual single-module runs shouldn't move the global cursor.

If the state file does not exist, this is fine — the plugin may not be initialized; just skip.

### Step 7: open the pull request
Follow `${CLAUDE_PLUGIN_ROOT}/skills/document-init/references/pr-flow.md`. In short:
1. `git checkout -b business-docs/module-<slug>-<YYYYMMDD-HHMM>`
2. `git add docs/modules/<slug>.md docs/glossary.md docs/_AUTOGENERATED.md`
3. Commit: `docs: regenerate <slug> module`
4. `git push -u origin <branch>`
5. `gh pr create` (or REST API with `GITHUB_TOKEN`).
6. PR body lists what was regenerated, counts, top `{{TBD}}` fields.

If no credentials, fall back to local-only and print manual push/PR steps.

### Step 8: report
Print:
- **PR URL** or local path + manual steps
- File regenerated (path)
- Counts: capabilities, external systems, business rules, risks
- Top 3 `{{TBD}}` fields most worth a human review
- ADR candidates surfaced (if any)

## Rules

- **No questions to the user.** Mark `{{TBD}}` and move on.
- **Diff-merge, don't blow away user edits.**
- **Always open a PR** when credentials allow.
- **Set `last-reviewed: <today>`** on regenerated files.
- **Lint Mermaid before writing** — parens in flowchart edge labels and colons in journey step text both produce silent render failures. Use em-dashes.
- Operate in English by default.
