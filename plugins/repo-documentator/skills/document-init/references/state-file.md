# `docs/.business-docs-state.json` — schema and rules

Plugin-managed state file. **Never edit by hand.** Used by `document-refresh` as the cursor for incremental updates.

## Why this exists

After `document-init` runs, the docs reflect the codebase **as of a specific commit**. Future runs of `document-refresh` need to know that commit so they can compute `git log <cursor>..HEAD` and process only what changed. The cursor lives in this file.

## Why not in `docs/README.md`

The README is curated by humans. Plugin-state belongs in a plugin-managed file or it will collide with user edits.

## Why hidden (`.` prefix)

Same reason `.gitignore`, `.eslintrc`, etc. are hidden — it's tooling configuration, not documentation. The visible summary lives in `docs/_AUTOGENERATED.md`.

## Schema

```json
{
  "$schema": "https://singular-innovation.com/business-docs/state-v1.json",
  "plugin": "business-docs",
  "plugin_version": "0.4.0",
  "schema_version": 1,
  "repo": {
    "remote_url": "https://github.com/<org>/<name>",
    "default_branch": "main"
  },
  "last_run": {
    "skill": "document-init | document-refresh | document-module | document-workflow | document-connections",
    "commit_sha": "<full 40-char SHA>",
    "commit_short": "<7-char SHA>",
    "commit_date": "<YYYY-MM-DD>",
    "commit_subject": "<first line of commit message>",
    "branch": "<branch the analysis ran against>",
    "ran_at": "<ISO 8601 UTC timestamp>",
    "ran_by": "business-docs plugin"
  },
  "modules": [
    {
      "slug": "test-runs",
      "paths": ["lib/test-runs/", "app/api/projects/[projectId]/test-runs/", "scripts/004_test_runs.sql", "..."],
      "last_regenerated_commit": "<SHA when this specific module was last regenerated>",
      "last_regenerated_at": "<ISO 8601>"
    }
  ],
  "workflows": [
    {
      "name": "agent-driven-test-run",
      "entry_points": ["app/api/v1/agent/auth/route.ts", "..."],
      "last_regenerated_commit": "<SHA>",
      "last_regenerated_at": "<ISO 8601>"
    }
  ]
}
```

## Field semantics

| Field | Purpose |
|---|---|
| `plugin_version` | Tracks which version of the plugin produced the state. If a future plugin version changes the doc structure, `document-refresh` can detect a stale state and decide to do a full regen. |
| `schema_version` | Schema evolution. Bump when fields are added/removed in a way that needs migration. |
| `repo.remote_url` | Helps catch the case where someone moved the docs to a different repo without re-init. |
| `repo.default_branch` | The branch the analysis ran against. Refresh refuses to run if the current branch differs. |
| `last_run.commit_sha` | **The cursor.** `document-refresh` uses this as `<cursor>..HEAD`. |
| `last_run.skill` | Which skill produced this state — useful for debugging. |
| `modules[].paths` | The path globs that belong to this module. Used to map changed files → affected module. |
| `modules[].last_regenerated_commit` | Per-module cursor — lets refresh skip modules whose paths haven't changed even if other parts of the repo moved. |
| `workflows[].entry_points` | The entry points used to derive the workflow. If any of these changed, the workflow doc may need re-derivation. |

## Lifecycle

### Written by `document-init`
After all docs are written and committed (step 8 of `document-init`), but **before** opening the PR (step 9), write the state file with:
- `last_run.skill = "document-init"`
- `last_run.commit_sha = HEAD` (the commit the analysis was based on)
- One entry per module (with paths from module-detector)
- One entry per workflow (with entry points from the trace)
- All `last_regenerated_commit` = same as `last_run.commit_sha`

Include the state file in the same PR.

### Read by `document-refresh`
First action: read `docs/.business-docs-state.json`. If absent, the repo wasn't initialized — skill stops with a clear error pointing to `document-init`.

### Updated by `document-refresh`
After regenerating affected modules/workflows:
- `last_run.skill = "document-refresh"`
- `last_run.commit_sha = HEAD`
- For each module that was actually regenerated → update `last_regenerated_commit`
- For each module that was checked but didn't need a refresh → leave `last_regenerated_commit` unchanged
- Same logic for workflows

Commit the state file along with the doc changes.

### Updated by `document-module` and `document-workflow`
When the user manually targets a single module or workflow:
- Update only that module/workflow entry's `last_regenerated_commit`
- Leave `last_run` pointing at the highest cursor (don't move backwards if a manual run was for an older state)

## Failure modes

### `commit_sha` no longer exists in repo
Repo was force-pushed or rebased. `git log <sha>..HEAD` fails.
**Behavior:** `document-refresh` warns the user, then falls back to comparing the file tree (path-based diff) and produces a full regen of any module whose files differ structurally. Updates the state to a new clean cursor.

### `default_branch` changed
**Behavior:** Refuse to run on a non-matching branch. Tell the user to either switch branch or re-init.

### `plugin_version` is older than the running plugin and the schema has incompatibilities
**Behavior:** Migrate the state file in place using a versioned migration map. If migration fails, instruct the user to re-init.

### State file is missing entirely
**Behavior:** Skill stops with an error pointing to `document-init`. Don't try to recover — re-init is the correct response.

## Visible summary

`document-init` and `document-refresh` both write a one-line summary into `docs/_AUTOGENERATED.md` so a reviewer can see the cursor without opening a hidden file:

> **Analyzed up to commit `<short-sha>` (`<commit-date>` — "<commit-subject>") on branch `<branch>`.**

Update this line on every refresh run.
