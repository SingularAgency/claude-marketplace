# document-init

Autonomous, non-interactive documentation generator. Given a GitHub URL (or local path), produces the full `docs/` skeleton, fills everything inferable from the code, and **opens a pull request** so the user reviews the docs as a normal code review.

## Core principles

- **No questions to the user. Ever.** Read the code, infer what can be inferred, mark the rest as `{{TBD}}`, push as a PR.
- **Non-technical by default.** The output must be readable by a PM or business stakeholder. No path:line references, no endpoint patterns, no SQL outside the Data Model section, no code blocks in business sections.
- **Diagrams in two layers.** Each module gets ONE business-friendly flow diagram. Each workflow gets TWO diagrams — a business view (flowchart) and a technical sequence (for engineers).
- **One file per module, not five.** Collapse `purpose / actors / business-rules / connections / known-pain` into a single `modules/<slug>.md` per module.
- **Diff-merge, don't blow away user edits.** A field already filled and not flagged `auto-detected: true` is user content — preserve it.

## Inputs accepted
- A GitHub URL (e.g. a repo URL, optionally with a branch path)
- A local path to an already-cloned repo

## Required for full operation
- Either `gh` CLI authenticated (`gh auth status` clean), or a `GITHUB_TOKEN` env var with `repo` scope, or a tokenized clone URL.
- Without credentials, the skill falls back to local-only mode and prints the manual steps for the user.

## Flow

### Step 1: resolve the repo source
**If a GitHub URL was provided:**
```bash
TARGET="./.business-docs-cache/<repo-name>"
git clone --depth=1 <url> "$TARGET"
# if URL contains /tree/<branch>, add: --branch <branch>
# Then: git fetch --unshallow  (so business-rules can pre-fill Owner from git first-commit author)
```
- For a public repo this just works.
- For a private repo without a token, the clone fails. Stop and tell the user the repo is private and they need to either provide a tokenized URL or `GITHUB_TOKEN`. **This is the only situation where you stop and ask.**

**If a local path was provided:** verify it exists. Continue.

### Step 2: pre-flight scan
Without writing anything yet:
- Detect language(s), framework(s), build system, repo shape (single project / monorepo / microservices / library / CLI).
- **Read existing docs** — `README.md`, anything under `docs/` (e.g. PRD, architecture notes), `CONTRIBUTING.md`, `CODEOWNERS`. These are gold for filling Purpose, Glossary, Stakeholders, and rule origins.
- Read `package.json` / `pyproject.toml` / `go.mod` / `Cargo.toml` / `pom.xml` etc. to extract project name, description, dependencies.
- Check whether `docs/` already exists. If it does, write the new content under `docs/_proposed/` instead of overwriting. **Do not delete or modify existing docs.**
- Get the top contributors: `git log --format='%an' | sort | uniq -c | sort -rn | head -10` — used to pre-fill `Owner` per module.

### Step 3: detect modules
Invoke the `module-detector` agent.

```
Agent({
  description: "Detect business modules in repo",
  subagent_type: "module-detector",
  prompt: "Analyze the repository at {path}. Propose the split into business modules (bounded contexts). Report using the format defined in your instruction."
})
```

Take the agent's output as the working module list — do not ask the user to confirm. Modules with low confidence get a `confidence: low` flag in their frontmatter so the user can review later.

### Step 4: create the global docs skeleton
Write the structure inside the repo (or under `docs/_proposed/` if `docs/` already existed):

```
docs/
├── README.md              # Business map + module table + workflow table
├── glossary.md            # Domain terms detected from APIs / models, definitions sourced from PRD where possible
├── stakeholders.md        # Personas from existing PRD; tech contributors from git log
├── connections.md         # Global connection map + Mermaid diagram + dependency matrix + subtle couplings (THIS GOLD)
├── workflows/             # One file per critical workflow (filled in step 6)
├── decisions/             # ADR templates (decisions are not auto-created — humans author them)
├── _AUTOGENERATED.md      # Single review checklist
└── modules/<slug>.md      # ONE file per module — see template
```

### Step 5: write per-module docs (one file each)
For each module from step 3, write `docs/modules/<slug>.md` following `${CLAUDE_PLUGIN_ROOT}/templates/module.md`. Sections (in order):

1. **Frontmatter** — `module`, `last-reviewed`, `status: draft`, `auto-detected: true`, `owner: <git first-commit author>`, `paths: [...]`.
2. **Purpose** (2-3 paragraphs, business language) — pull from PRD / repo README. NOT a one-liner.
3. **Module flow** (Mermaid `flowchart LR` with persona + outcome colors) — see Diagram conventions below.
4. **Capabilities** — bullets in plain English. **NO file paths, NO line numbers, NO endpoint patterns.** Examples:
   - GOOD: "Author a new test case with title, description, preconditions, steps, priority, and tags."
   - BAD: "Create test case via `POST /api/projects/:projectId/test-cases` (`app/api/.../route.ts:74`)".
5. **Data Model** — tables / columns / FKs / indexes. **This section IS allowed to be technical.** Cite migration files.
6. **Connections** (this module ↔ others) — what it consumes, exposes, publishes, subscribes. Cite path:line. Pull the relevant subset from the global `connections.md` inline.
7. **Business Rules** — each rule has `Rule:`, `Owner:` (auto-filled from git first-commit author with verify caveat), `Origin:` (commit reference). **No `Implementation:`, `Revisable:`, or `Exceptions:` lines.** Statement in policy language, not code.
8. **Actors** — Human actors (real role inferences with role-check evidence), External systems (be specific), Other modules.
9. **User journey maps** — one Mermaid `journey` diagram per real actor of the module. Each shows the actor's steps with realistic satisfaction scores (0-5). NO colons inside step text — Mermaid splits on colons. Use em-dashes inside step text.
10. **Related workflows** — list of links to `workflows/<flow>.md` with one-line role per workflow. NO diagrams duplicated here.

#### Heuristics for auto-derivation
- **Purpose**: pull from repo README, PRD, package description. Only invent if all three are silent → mark `{{TBD}}`.
- **Capabilities**: list public API routes, exported functions, CLI commands, UI affordances — **restated in business language**. Strip every endpoint/path before writing.
- **Data Model**: full table schemas with constraints. Cite migration filename (this is the one place migration refs are appropriate).
- **Connections**: cross-module imports, HTTP/RPC calls, events, queues, shared DB, env vars. Cite path:line for evidence.
- **Business Rules**: numeric thresholds, role checks, status transitions, lifecycle invariants. Pre-fill `Owner` with git first-commit author + caveat. Pre-fill `Origin` with `introduced in <sha> by <author> on <date> — "<commit subject>"`.
- **Actors**: external systems detected from imports / SDKs / env vars; human roles inferred from route guards / role checks.
- **User journey maps**: for each real actor identified in Actors, write one Mermaid `journey` diagram showing their steps + realistic satisfaction scores. Tell a story with peaks and valleys; identify pain points based on the code (e.g. multi-step uploads → score 3, slow async waits → score 2, instant feedback → score 5).

### Step 6: derive critical workflows
Identify 2-5 critical end-to-end flows from the entry-point trace (the most-called API routes, the most-bridged events). For each one, derive the workflow doc following `${CLAUDE_PLUGIN_ROOT}/templates/workflow.md`. Sections:
1. Business objective + Trigger + Actors involved
2. **Business view (non-technical)** — Mermaid `flowchart LR` with persona + outcome colors. Plus one paragraph each: "What this is", "Who's involved", "Why it matters".
3. **Technical sequence (for engineers)** — Mermaid `sequenceDiagram` with `autonumber`, all human actors AND systems as participants, full endpoint / DB / event detail.
4. Variants (if applicable)
5. **Happy path** — numbered steps in plain English. NO `[API → DB]` notation, NO path:line refs.
6. **Edge cases** — each one in `What it looks like to the user / Why / Resolution` format, business framing.
7. Business rules applied — links to `modules/<x>.md` with rule restated in plain language.
8. Key data flowing (table) — plain English, not schemas.
9. SLAs / Metrics / Incident history — TBDs are honest where production data isn't available.

### Step 7: run connection analysis
Invoke `connection-analyst` for the global `docs/connections.md`.

```
Agent({
  description: "Map inter-module connections",
  subagent_type: "connection-analyst",
  prompt: "Analyze the connections between modules {list}. Generate the global map and per-module relevant subsets. Cite evidence (path:line) for every connection. Flag subtle couplings (shared DB tables, shared env vars, shared storage, implicit payload conventions) with emphasis."
})
```

Insert the global graph into `docs/README.md` as the module map. Pull each module's relevant subset into its `Connections` section.

### Step 8: post-process
- Set `last-reviewed: <today>` on every generated file.
- Set `status: draft` and `auto-detected: true` everywhere applicable.
- **Lint Mermaid blocks** — two known foot-guns:
  1. **Parens inside `|...|` edge labels in flowcharts** — Mermaid parses `(` as a node-shape delimiter and the diagram breaks. Replace with em-dashes: `|Manual — QA Engineer|`.
  2. **Colons inside step text in `journey` diagrams** — Mermaid splits `step: score: actor` on the colons. A step description containing `:` confuses the parser. Replace inline colons with em-dashes inside the step text.
- Write `docs/_AUTOGENERATED.md` with: counts (modules / connections / rules / journeys), top `{{TBD}}` fields by impact, ADR candidates, file inventory, and the **cursor line** (see below).
- **Write the state file** `docs/.business-docs-state.json` with the cursor (current HEAD), per-module and per-workflow paths, plugin version. See `${CLAUDE_PLUGIN_ROOT}/skills/document-init/references/state-file.md` for the exact schema. This is what `document-refresh` reads to do incremental updates.
- **Add the visible cursor line** at the top of `docs/_AUTOGENERATED.md`:
  > **Analyzed up to commit `<short-sha>` (`<date>` — "<subject>") on branch `<branch>`.**
  This line is updated on every subsequent refresh run.

### Step 9: open the pull request
Follow `${CLAUDE_PLUGIN_ROOT}/skills/document-init/references/pr-flow.md`. Summary:
1. `git checkout -b business-docs/init-<YYYYMMDD-HHMM>`
2. `git add docs/` (this includes `docs/.business-docs-state.json`)
3. Commit: `docs: initial business documentation (auto-generated)`
4. `git push -u origin <branch>`
5. **Prefer `gh pr create`**. Fall back to GitHub REST API with `GITHUB_TOKEN` if `gh` is unavailable.
6. PR body = contents of `docs/_AUTOGENERATED.md` + auto-gen footer.
7. Print the PR URL to the user.

If credentials are missing, write the docs locally and tell the user exactly how to push and open the PR manually.

### Step 10: report
Print:
- **PR URL** (the deliverable) — or local path + manual steps if no creds
- Repo source (URL or local path)
- Module count, workflow count, connection count, business-rule count, risk count
- Top 5 `{{TBD}}` items the user should fill first

## Diagram conventions

### Module flow (per-module flowchart)
- One `flowchart LR` per module file, between Purpose and Capabilities.
- 5-10 nodes maximum. Show the typical user interaction with this module.
- Use these classDefs:
  - `:::person` (blue) → human actors
  - `:::good` (green) → positive outcomes
  - `:::warn` (amber) → cautionary states / queued / pending
  - `:::neutral` (grey) → terminal-but-not-bad states
- **NEVER use parentheses inside `|...|` edge labels.** Use em-dashes. Mermaid's parser interprets `(` as a node-shape delimiter and the diagram fails to render.

### Workflow business view (cross-module flowchart)
- `flowchart LR` at the top of each workflow file.
- Same color convention as module flow.
- Plain-English nodes only. NO endpoints, NO databases, NO internal services.

### Workflow technical sequence (cross-module sequence diagram)
- `sequenceDiagram` with `autonumber`.
- Include human actors AND systems as participants.
- Cite endpoints, DB writes, events, queues — this is the engineer-friendly view.

## Rules

- **Never ask the user a question except when the repo is private and unauthenticated.** Everything else is inferred or marked `{{TBD}}`.
- **Always open a PR** when credentials allow. The PR is the deliverable, not the local files.
- **Don't overwrite existing `docs/`.** Always write to `docs/_proposed/` if a previous `docs/` exists.
- **Never push to the default branch.** Always a `business-docs/<skill>-<timestamp>` feature branch.
- **Never force-push.** Conflicts → new branch with a numeric suffix.
- **Never include credentials** in commits, PR bodies, or remote URLs in output.
- **Lint every Mermaid block before writing** — two failure modes: parens in flowchart edge labels, and colons inside step text in `journey` diagrams. Both produce silent render failures on GitHub. Use em-dashes in both cases.
- **Idempotent:** running twice on the same repo produces the same docs (modulo `last-reviewed`); a fresh run opens a new PR.
- Operate in English by default (unless the repo's README or comments are clearly in another language).
