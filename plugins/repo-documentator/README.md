# business-docs

A Claude Code plugin that **autonomously documents a repository** given a GitHub URL and **opens a pull request** with the generated docs. The output is **non-technical, PM-friendly** business documentation — readable by stakeholders, sales, customer success, and developers alike.

## Philosophy

The AI can already read the code. **No interviews, no questions to the user.** What can be derived is auto-derived. What only a human knows is marked `{{TBD}}` so reviewers fill it during PR review.

Two design principles drive the output:

1. **Non-technical by default.** Capabilities, Business Rules, and Known Risks read in plain English. No file paths, no line numbers, no SQL outside the Data Model section, no internal endpoint mentions in business sections.
2. **Diagrams in two layers.** Each module gets a single business-friendly flow diagram. Each cross-module workflow gets two — a business view (flowchart with persona + outcome colors) and a technical sequence (with endpoints, DB, events for engineers).

## How it works

### Initial run

```
You: "Document <github-url>"
Plugin:
  1. Clones the repo (shallow, read-only)
  2. Runs module-detector (subagent) → bounded contexts
  3. Runs connection-analyst (subagent) → connection map + subtle couplings
  4. Reads the code AND existing docs (PRD, README, CODEOWNERS) to fill business context
  5. Writes one cohesive markdown file per module + per workflow
  6. Records the analysis cursor (commit SHA + per-module paths) in docs/.business-docs-state.json
  7. Marks human-only fields as {{TBD}} with a single review checklist
  8. Commits to a feature branch + opens a pull request — that's the deliverable
```

### Subsequent incremental runs

```
You: "Refresh the docs"
Plugin (document-refresh):
  1. Reads the cursor from docs/.business-docs-state.json
  2. Computes git log <cursor>..HEAD to find changed files
  3. Maps changed files to affected modules / workflows
  4. Regenerates ONLY the affected ones (untouched modules keep their last-reviewed date)
  5. Diff-merges against any user edits in those files
  6. Advances the cursor to the new HEAD
  7. Opens a PR with the drift report as the body
```

Modules whose paths didn't change are skipped entirely — no needless rewrites, no merge noise. Each module also has its own `last_regenerated_commit`, so refresh is granular.

## Output structure

```
docs/
├── README.md              # Project map + module table + workflow table
├── glossary.md            # Domain terms (definitions sourced from PRD where possible)
├── stakeholders.md        # Personas from PRD; tech contributors from git log
├── connections.md         # Global technical connection map + subtle couplings
├── workflows/             # One file per critical end-to-end flow (with 2 diagrams)
├── decisions/             # ADR templates (decisions are not auto-created — humans author them)
├── _AUTOGENERATED.md      # Single review checklist of every TBD + ADR candidate
└── modules/               # ONE file per module — see structure below
    ├── <slug-1>.md
    └── <slug-2>.md
```

## Per-module file structure

Each `modules/<slug>.md` has these sections in this order:

| Section | Audience | Technical? |
|---|---|---|
| Frontmatter | Tooling | – |
| Purpose | PMs, stakeholders | Plain English (2-3 paragraphs) |
| **Module flow** | PMs, stakeholders | Mermaid flowchart with persona + outcome colors |
| Capabilities | PMs, stakeholders | Plain English bullets — NO paths/lines/endpoints |
| Data Model | Engineers | Tables, columns, FKs, constraints, indexes |
| Connections | Engineers | Cross-module imports, calls, events with path:line |
| Business Rules | PMs + Engineers | Rule (plain English) + Owner (auto from git) + Origin (commit ref) |
| Actors | PMs + Engineers | Human roles + external systems + other modules |
| **User journey maps** | PMs, UX, leadership | One Mermaid `journey` diagram per actor — steps with satisfaction scores (0-5) |
| Related workflows | All | List of links to `workflows/<flow>.md` |

## Per-workflow file structure

Each `workflows/<flow>.md` has these sections in this order:

| Section | Audience | Technical? |
|---|---|---|
| Frontmatter | Tooling | – |
| Business objective | All | Plain English |
| Trigger | All | Plain English |
| Actors involved | All | Humans + systems |
| **Business view (non-technical)** | PMs, stakeholders | Mermaid flowchart with persona + outcome colors |
| **Technical sequence (for engineers)** | Engineers | Mermaid sequence diagram with endpoints, DB, events |
| Variants | All | Branch points |
| Happy path | All | Numbered steps in plain English (no `[API → DB]` notation) |
| Edge cases | All | What it looks like to the user / Why / Resolution |
| Business rules applied | All | Cross-refs to `modules/<x>.md` rules |
| Key data flowing | All | Plain-English data table |
| SLAs / Metrics / Incident history | All | Honest TBDs where production data isn't available |

## Setup — credentials for opening PRs

The plugin needs to push and open a PR. It checks, in order:

1. **`gh` CLI authenticated** — preferred. Run `gh auth login` once.
2. **`GITHUB_TOKEN` env var** with `repo` scope — used for cloning private repos and the PR API call.
3. **Tokenized clone URL** passed to `document-init` (`https://<token>@github.com/...`).
4. **None of the above** → falls back to writing locally and prints manual push/PR steps.

For private repos: option 2 or 3 is required for the clone to even work.

## Available skills

| Skill | What it does | PR title prefix |
|---|---|---|
| `document-init` | Clones a repo and opens a PR with the full `docs/` skeleton | `docs: initial business documentation` |
| `document-module` | Regenerates a single module's doc from the code | `docs: regenerate <name> module` |
| `document-workflow` | Traces a workflow from an entry point and writes a 2-diagram workflow file | `docs: derive <name> workflow` |
| `document-connections` | Regenerates the inter-module connection map | `docs: regenerate inter-module connection map` |
| `document-refresh` | Detects drift and opens a PR with the refresh | `docs: refresh — N modules, M connections` |

Every skill that writes to `docs/` opens its own PR. Branches are named `business-docs/<skill>-<YYYYMMDD-HHMM>`.

## Conventions

- Every auto-derived file carries `auto-detected: true` in frontmatter — `document-refresh` rewrites those without touching user edits.
- Files **without** `auto-detected: true` are user-owned. The plugin never overwrites them.
- Every auto-detection in technical sections (Data Model, Connections) cites code evidence (`path:line`).
- Every Mermaid block is linted before write. Two known foot-guns:
  1. **Parens inside `|...|` edge labels** in flowcharts (Mermaid's parser breaks on them) — use em-dashes.
  2. **Colons inside step text** in `journey` diagrams (Mermaid splits step lines on `:`) — use em-dashes inside step text.
- Re-running on the same repo is idempotent (modulo `last-reviewed` dates).
- **Never pushes to the default branch.** Always a feature branch + PR.
- **Never force-pushes.** Conflicts → new branch with a numeric suffix.

## Plugin state file

`docs/.business-docs-state.json` is plugin-managed (do not edit by hand). Tracks:
- Plugin version + schema version (for migrations)
- Repo + default branch
- The analysis cursor (commit SHA, date, subject, branch)
- Per-module: paths + last regenerated commit
- Per-workflow: entry points + last regenerated commit

`document-refresh` reads this file to know what to compare against. The visible cursor line in `docs/_AUTOGENERATED.md` mirrors the cursor for human reviewers.
